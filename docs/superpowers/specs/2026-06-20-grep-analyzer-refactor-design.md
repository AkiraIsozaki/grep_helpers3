# grep_analyzer コード健全性リファクタリング 設計（診断ファースト）

- 日付: 2026-06-20
- 対象: `src/grep_analyzer/`（57ファイル / 約5,400 LOC）
- 目的: コードベース全体の健全性向上。**命名・構造・シンプル化（重複/YAGNI）を同格の軸**として扱い、
  保守性向上を図る。命名は重要な関心事だが、AI レビューでのサンプリング所見（命名は概ね健全で
  明白な命名臭は局所的）を踏まえ、唯一の最優先には据えず重みを再配分する。

## 背景と狙い

`grep_analyzer` は grep/ripgrep 出力を起点にシンボルを追跡・分類する解析ツール。
すでにモジュール分割は進んでいる（最大 `fixedpoint/_scan.py` で 422 行）が、ユーザーの観察として
「プログラム中に長めのコメントが書かれていることがままあり、設計（特に名前）がうまく
いっていないサインではないか」という懸念がある。

本作業の方針:

- 長いコメントは **「名前・構造で語れていない代償」のサイン**として扱う。
  コメント削除を目的化せず、**名前付け・関数分割でコードを自明化し、結果としてコメントが減る**ことを狙う。
- 正当な WHY/契約（非自明な判断、不変条件、降格契約など）を説明するコメントは**残すべき価値**として区別する。
  例: `fixedpoint/_seed.py` の「seed_hits は direct パスがファイル単位でまとめて構築するため
  同一ファイルが連続する」は正当な WHY。
- メモリ記載の方針を順守する: コメントは保守的にトリム / コメントは体言止めを避け述語のある文 /
  main 直コミット（feature branch なし） / pytest は wheelhouse の `.venv` を有効化して実行。
  **baseline は実走で確定した `739 passed, 10 skipped`（2026-06-20 時点）を固定基準とする。**
  退行検出のため「738 でも 638 より多いから OK」のような相対判断はせず、絶対値で照合する。

## フェーズ構成

**フェーズ1（read-only 診断）→ フェーズ2（TDD で選択的に実装）** の二段構え。
コードを変更するのはフェーズ2、診断レポートにユーザーが合意した後のみ。

## フェーズ1: 診断

### スキャン方法

パッケージ単位で並列に監査を行い、横断的に所見を集約する。`src/grep_analyzer/vendor/`
（ripgrep バイナリ。Python ソースなし）は監査対象外とし、LOC 計数にも含めない。区分け:

- **top-level**: `cli`, `pipeline`, `dispatch`, `walk`, `ripgrep`, `ingest`, `resume`,
  `output_writer`, `model`, `chase`, `decode_cache`, `budget`, `spill`, `provenance`,
  `tsv`, `encoding`, `automaton`, `stoplist`, `classify`, `progress`, `diagnostics`,
  `embed_preprocess`, `proc_preprocess`
- **fixedpoint/**: `_scan`, `_lockstep`, `_seed`, `_ingest`, `_state`, `_finalize`,
  `_options`, `_budget_control`, `_encmemo`
- **classifiers/**: 各言語 chaser（`python_chaser`, `java_chaser`, `c_chaser`,
  `sql_chaser`, `perl_chaser`, `shell_chaser`, `groovy_chaser`, `javascript_chaser`,
  `typescript_chaser`）+ `base`, `regex_classifier`, `ts_classifier`
- **snippet/** + **patterns/**: `_heuristic`, `_clamp`, `_ts`, `_sanitize_line` /
  `symbol_extraction`, `snippet_boundaries`, `literal_masking`

### 診断カテゴリ

各所見は次を持つ: **場所** `file:line` / **問題** / **重大度**（high/medium/low） /
**労力**（S/M/L） / **改善案** / **before→after スケッチ**。

命名・構造・シンプル化は同格に扱う。

1. **命名** — 自明でない関数/変数/モジュール名、略語、誤解を招く名前、
   モジュール間の一貫性欠如、ドメイン語彙との不一致。
2. **構造・責務** — 1関数が複数責務を持つ、長すぎる関数、曖昧なモジュール境界、
   不適切な依存方向。
3. **シンプル化・重複** — 重複ロジック、過剰な分岐、不要な間接、YAGNI 違反、
   デッドコード。
4. **コメント臭** — 「名前/構造で語れていないための長コメント」を特定し、
   命名・分割での自明化案に変換する。**各所見で「設計の代償」か「正当な WHY/契約」かを
   個別に根拠付ける**こと。AI レビューのサンプリングでは長コメントの多数が並行処理の
   不変条件・キャッシュ無効化契約・TOCTOU 降格契約などの**正当な WHY であった**ため、
   契約・不変条件を説明するコメントはデフォルトで「**残す**」と判定する。

### 成果物

`docs/superpowers/specs/2026-06-20-grep-analyzer-refactor-findings.md` に診断レポートを出力する。

- 所見ごとに一意 ID を採番する。並列監査での衝突を避けるため、カテゴリとパッケージを
  含めた名前空間で振る（例 `NAME-fixedpoint-01`, `SIMPL-classifiers-02`, `CMT-snippet-01`）。
- **優先順位の合成規則**: 各所見を「重大度（high>medium>low）× 労力（S>M>L、小労力ほど高 ROI）」で
  評価し、**ROI（重大度の高さ ÷ 労力の大きさ）降順**で並べる。同点は重大度を優先する。
  冒頭にこの規則で並べた優先順位表を置く。
- 各所見に before→after の具体スケッチを付け、合意の単位を明確にする。
- 各所見に**ステータス欄**（未着手 / 即修正済 / 合意待ち / 実装済 / 見送り）を設け、
  フェーズ2の進捗・中断再開はこのレポート上で追跡する。
- 所見が多数に上る場合、レポート本体には high/medium を載せ、low は付録にまとめて
  合意プロセスの肥大を防ぐ。

## フェーズ2: 実装

### 二経路（オーバーヘッド削減）

- **S 級の局所修正**（1文字変数のリネーム、明白なデッドコード除去など、影響範囲が局所で
  低リスク）は診断レポートでの個別合意を経ず、即 TDD で修正してよい。レポートには
  「即修正済」として事後記録する。
- **M/L 級の所見**（モジュール境界の変更、広域リネーム、重複の共通化など）はレポートに
  載せ、ユーザーが着手項目を選択（または「上位から」「全部」等の指示）してから実施する。

### 実施ルール

- 各項目を **TDD**（test-driven-development スキル）で実施。挙動を変えない純粋
  リファクタを原則とする。
- **純粋リファクタの境界**: 挙動・出力を変えない限り、リネームに伴う**テストの import/参照の
  機械的追随は純粋リファクタに含む**（相談不要）。これは「テストの意味的変更」（期待値・
  アサーションの変更を要する＝挙動変更）とは区別する。後者のみ挙動変更として相談する。
  ※ 29 個のテストファイルが内部 `_` シンボルを直接 import しているため、内部リネームは
  ほぼ必ずテスト参照の更新を伴う。
- **広域リネームの安全手順**: 参照箇所は pytest だけに頼らず、`grep` で旧名（文字列参照・
  `getattr`/動的解決・ドキュメント内の旧名を含む）を洗い出してから一括更新する。
- コメントは保守的に扱う（メモリ方針順守）。
- 挙動変更が必要な所見は、純粋リファクタと分離し、都度フラグを立ててユーザーに相談する。
- **コミット粒度**: 1 所見＝1 コミットを基本とする。広域リネームは参照更新まで含めて
  1 コミットにまとめ、レビュー可能な単位を保つ。まとまった論理単位ごとに main へ直コミットする。

## 検証

- 各変更後に pytest 全件 green を確認する（`.venv` を wheelhouse から有効化、
  baseline `739 passed, 10 skipped` を絶対値で照合）。
- **安全網**: golden スイート（フルパイプライン TSV 完全一致, 49 ケース）が純粋リファクタの
  退行検出の主軸であり、強力に機能する。カバレッジ計測は本作業では必須としない（導入は任意）。
- `requires_ripgrep` 系テストは vendor 配置に依存する。vendor 未配置の環境では skip 数が
  増えるため、baseline 照合は vendor を配置した同一環境で行う。
- 純粋リファクタ原則のため、テストの意味的変更（期待値・アサーション）が必要になった場合は
  リファクタではなく挙動変更として扱い相談する。

## 完了の定義

- レポートの全所見が「実装済」または「見送り」のいずれかに確定している。
- baseline テスト（`739 passed, 10 skipped`）が維持されている。
- 「全部直す」ことは完了条件ではない。見送り判定が明示されていれば完了とみなす。

## 非目標（YAGNI）

- 新機能の追加。
- 挙動・出力フォーマットの変更（必要が判明した場合は別途相談）。
- 今回のゴールに無関係な大規模アーキテクチャ刷新。
